# 产品需求文档 (PRD)：具备多级记忆与反思机制的终身学习学术 Agent

## 1. 项目概述

### 1.1 背景与痛点

在复杂科学与学术网络（如生物医药网络、社交网络演进、GNN 架构设计）的研究中，科研人员面临的核心痛点是：

- **上下文碎片化：** 大模型（LLM）在单次 Session 结束后状态清零，无法跨越数周或数月的科研周期记住研究员的独特观点和纠错结论。
- **学术事实幻觉：** 复杂的跨网络关联（如跨模态图卷积、多关系图注意力机制）极易引发大模型的语义混淆。
- **无法自适应进化：** 当用户纠正了 Agent 的某项学术推导错误后，在下一次对话或切换概念时，Agent 依然会犯同样的错误。

### 1.2 核心价值主张

本项目构建一个具备“工作-情节-语义”三级记忆架构**的终身学习智能体。通过**用户显式反馈（Explicit Feedback）驱动的反思机制，Agent 能够从学术交互失败中提取“硬核规则（Guidelines）”，永久沉淀至语义知识图谱中，实现真正的学术协同进化。

------

## 2. 核心业务场景定义

Agent 优先落地于**复杂科学与学术网络研究分析场景**，具体包括但不限于：

- **学术概念与文献关系推导：** 例如分析“异质图神经网络（HGNN）在药物-疾病关联（DDA）预测中的最新进展”，梳理不同算法（如 ATDR, MAGCN, TARSL）的演进关系。
- **技术路线可行性论证：** 评估特定图注意力转移机制在多模态数据融合中的逻辑漏洞。

------

## 3. 系统架构与功能需求

### 3.1 三级记忆架构 (Multi-Level Memory Architecture)

Agent 必须协同调度以下三层记忆组件：

| **记忆层次**                   | **物理载体**                                        | **存储内容**                                                 | **作用域与生命周期**                              |
| ------------------------------ | --------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------- |
| **工作记忆 (Working Memory)**  | 内存 / Redis                                        | 当前会话的 Prompt 窗口、短期上下文、正在调用的工具输出（Tool Outputs）。 | 单次 Session，随进程结束或主动清空而销毁。        |
| **情节记忆 (Episodic Memory)** | 向量数据库 (Qdrant / Chroma 本地模式)               | 历史对话片段、用户的原始提问、Agent 的逐步推理链（CoT）及学术结论。 | 跨 Session 永久存储，通过 Embedding 相似度检索。  |
| **语义记忆 (Semantic Memory)** | 轻量级知识图谱 (NetworkX / LightRAG 本地 JSON 结构) | 学术实体（如算法、数据集、基因）、关系（如“改进自”、“优于”）、以及反思生成的**黄金指南（Guidelines）**。 | 跨 Session 永久存储，通过实体链接与拓扑关系检索。 |

```
[用户输入] ──> 检索 (工作记忆 + 情节记忆向量 + 语义图谱子图) ──> LLM 整合推理
```

### 3.2 显式反馈驱动的反思机制 (Reflection Loop)

这是系统的核心进化逻辑。当 Agent 输出的学术推导出现错误，用户给予显式否定反馈时，触发反思流。

#### 3.1.1 触发条件

用户输入包含强否定语义或特定指令（如：`"不对"`、`"逻辑错了"`、`"MAGCN 不是这样处理图注意力的"`、`"修改学术假设"` 等）。

#### 3.1.2 反思与写入流程

1. **冻结当前上下文：** 提取当前 Working Memory 中的最后 3 轮对话作为错误样本。

2. **调用反思链（Reflection CoT）：** 启动一个独立的 Prompt 模板（Critique LLM），分析错误根源：

   - *问题：* 为什么会产出这个学术错误？（是因为混淆了概念，还是遗漏了约束条件？）
   - *解法：* 正确的学术逻辑应当是什么？

3. **生成硬核指南（Guideline Extraction）：** 生成一条格式化的、普适性的规则。

   > **示例：** `[Guideline] 当处理多模态图卷积网络（MAGCN）的特征融合时，跨网络注意力层必须在特征对齐之后进行，不可直接对原始异质节点做点积。`

4. **更新语义图谱：**

   - 将该 Guideline 作为一个新的“规则节点（Rule Node）”插入语义记忆（KG）中。
   - 通过边（Edges）将该规则节点连接到相关的学术实体（如 `MAGCN`、`跨网络注意力`）。

### 3.3 记忆唤醒与推理流程 (Retrieval & Inference)

在下一次新对话中，Agent 的前置检索器（Retriever）按以下优先级组装 Context：

1. **实体提取：** 从用户输入中提取学术实体（如 "我准备用 MAGCN 做链路预测..."）。

2. **图谱关联唤醒：** 在语义记忆（KG）中查询该实体，**优先拉取与其连接的 [Guideline] 节点**。

3. **情节相似度唤醒：** 在情节记忆（Vector DB）中检索过去类似的讨论场景。

4. **Prompt 组装：**

   Plaintext

   ```
   【系统防错机制：你必须严格遵守以下学术审查规则】
   - $从历史反思中提取的 Guideline 1$
   - $从历史反思中提取的 Guideline 2$
   
   【相关历史背景】
   $情节记忆片段$
   
   【当前任务】
   $用户当前输入$
   ```

## 4. 接口与交互设计 (MVP 阶段)

考虑到使用 Claude Code 协作开发，首期放弃复杂的 Web UI，采用**轻量级 CLI（命令行接口）或标准 API**。

### 4.1 CLI 交互设计示例

Bash

```
User >> 请分析基于注意力的转移机制（TARSL）在异质生物网络中的局限性。
Agent >> [结合语义记忆中已有的 GNN 规则进行推导...] TARSL 主要局限在于它假设不同关系空间的注意力分布是各向同性的...
User >> 不对，TARSL 正好相反，它是通过关系特异性投影来实现各向异性的，你把投影方向搞反了。
Agent >> [触发显式反思机制...] 
         正在分析错误... 
         已成功提取 Guideline: "TARSL 采用关系特异性投影实现各向异性注意力机制"。
         已将此规则绑定至语义记忆中的 [TARSL] 与 [各向异性] 节点。
User >> 重新分析。
Agent >> [已自动载入最新 Guideline] 重新评估：在 TARSL 架构下，由于引入了关系特异性投影...
```

## 5. 非功能性需求与技术选型推荐

- **开发语言：** Python 3.10+ (利于科学计算、图网络处理和 Claude Code 快速迭代)
- **图处理库：** **NetworkX**（MVP 阶段直接序列化为本地 JSON 文本文件，简单可靠，便于 Claude Code 进行 debug）。
- **向量数据库：** **Qdrant (Local Memory mode)** 或 **ChromaDB**，无需配置复杂的 Docker 环境。
- **大模型接入：** 使用 Anthropic API (Claude 3.5 Sonnet / Claude 3 Opus) 以保证高强度的逻辑推理能力，尤其是反思和提取 Guideline 的质量。
- **可观测性：** 必须记录每次 Reflection 的日志（错误输入、生成 Guideline 的 Prompt、图谱变更前后对比），方便开发者人工 review 图谱是否被“污染”。

------

## 6. 开发任务分解与 Milestone (Claude Code 实施指南)

- **Phase 1：基础脚手架与三级记忆存储 (1-2天)**
  - 构建 CLI 基础交互框架。
  - 实现基于 Memory/JSON 的三级记忆读写接口（确保能 Save/Load）。
- **Phase 2：检索链与标准推理 (2天)**
  - 编写实体提取器（Entity Extractor），实现输入文本与 NetworkX 图节点的映射。
  - 实现融合“向量检索+图谱 Guidelines+当前上下文”的混合 Prompt 组装引擎。
- **Phase 3：反思状态机 (Reflection Pipeline) 核心开发 (2天)**
  - 编写用户否定词拦截器。
  - 编写 Reflection Prompt，专门负责从失败会话中蒸馏 `Guidelines`。
  - 实现 NetworkX 图动态插入新 Rule 节点和 Edge 的逻辑。
- **Phase 4：边界测试与健壮性对齐 (1天)**
  - 模拟多次学术逻辑反转，测试 Agent 是否会产生“记忆过拟合”或规则冲突。
